# Deployment Architecture

Network selection, contract deployment, and upgrade strategy for RevvFi.

---

## Network Selection

### Primary Networks

RevvFi v1 prioritizes Base and Arbitrum for optimal cost and accessibility.

| Network | Chain ID | Gas Price | Choice | Rationale |
|---------|----------|-----------|--------|-----------|
| **Base** | 8453 | ~1 Gwei | **Primary** | Low cost, growing ecosystem, Coinbase backing |
| **Arbitrum** | 42161 | ~0.5 Gwei | **Primary** | Ultra-low gas, high liquidity, mature ecosystem |
| **Ethereum** | 1 | ~50 Gwei | **Secondary** | High security, maximum decentralization, 10-50x cost |

### Network Characteristics

**Base:**
- EVM-equivalent rollup on Ethereum
- ~7-day withdrawal to mainnet
- Sequencer maintained by Coinbase
- Strong validator set
- Growing TVL and user base

**Arbitrum:**
- Optimistic rollup on Ethereum
- ~1-week challenge period for withdrawals
- Decentralized validator set
- Highest liquidity of L2s
- Mature ecosystem (1000+ projects)

**Ethereum:**
- L1 (mainnet)
- Final settlement
- Highest security
- Highest cost (10-50x more gas)
- Optional for conservative projects

## Contract Upgrades

### Upgrade Strategy

RevvFi uses **selective upgradability**: only protocol-level contracts upgrade, not per-launch contracts.

**Upgradeable Contracts:**
- RevvFiFactory
- RevvFiGovernance
- TokenTemplateFactory
- CreatorProfileRegistry
- PopularityOracle

**Non-Upgradeable Contracts:**
- RevvFiBootstrapper (per-launch)
- CreatorVestingVault (per-launch)
- TreasuryVault (per-launch)
- StrategicReserveVault (per-launch)
- RewardsDistributor (per-launch)

### Upgrade Process

#### Step 1: Propose Upgrade

```solidity
// Guardian multisig calls:
factory.proposeUpgrade(
    address newImplementation,
    string memory reason
);

// Emits: UpgradeProposed(newImplementation, reason, timestamp)
```

#### Step 2: Timelock Delay

- Upgrade locked for 7 days
- Allows community time to react
- Can be cancelled by governance vote

#### Step 3: Execute Upgrade

```solidity
// After 7 days, guardian executes:
factory.executeUpgrade(newImplementation);

// Implementation pointer updated
// New code takes effect immediately
```

### Rollback Plan

If upgrade causes issues:

```solidity
// Guardian can rollback to previous implementation
factory.rollback(previousImplementation);

// Reverts to known-good version
// Governance can then investigate
```

## Per-Launch Contract Lifecycle

### Contract Creation

When creator calls `createLaunch()`:

```
1. RevvFiFactory creates new instances:
   - RevvFiBootstrapper ──┐
   - CreatorVestingVault  ├─ All created in one TX
   - TreasuryVault        ├─ Immutable after creation
   - StrategicReserveVault│
   - RewardsDistributor ──┘

2. Factory stores references:
   - bootstrappers[launchId] = address
   - isDeployed[address] = true

3. Deploy token:
   - TokenTemplateFactory.deployToken()
   - Transfer allocations to vaults
```

### Contract Lifecycle

```
                   Created
                     │
                     ▼
              ┌──────────────┐
              │  DEPOSIT     │  LPs can deposit ETH
              │  PHASE       │  Shares minted
              └──────┬───────┘  Runs until raiseEndTime
                     │
     ┌───────────────┴───────────────┐
     │ (If target reached)           │ (If target NOT reached)
     ▼                               ▼
┌──────────────┐               ┌──────────────┐
│   LAUNCH     │               │    FAILED    │
│   PHASE      │               │   PHASE      │
└──────┬───────┘               └──────┬───────┘
       │ liquidityDeployed = true  │ LPs claim refunds
       │ Uniswap pair created      │ ETH returned
       ▼                               │
┌──────────────┐                      │
│  LOCK        │                      │
│  PERIOD      │                      │
└──────┬───────┘                      │
       │ Runs until maturityTime    │
       │ LPs cannot withdraw        │
       │ Governance active          │
       ▼                            │
┌──────────────┐                    │
│   MATURE     │                    │
│   PHASE      │                    │
└──────┬───────┘                    │
       │ LPs can withdraw           │
       │ withdrawAsAssets()      ←──┘
       │ withdrawAsETH()
       ▼
    [ENDED]
```

## Disaster Recovery

### Scenario 1: Factory Contract Compromised

**Response:**
1. Guardian pauses factory (`pauseFactory()`)
2. Existing launches unaffected
3. New launches blocked
4. Deploy new factory
5. Migration via governance vote

### Scenario 2: Per-Launch Contract Exploit

**Response:**
1. Guardian pauses affected bootstrapper
2. LPs cannot deposit/withdraw
3. Governance investigates
4. Manual resolution needed
5. Document for future audits

### Scenario 3: Uniswap Failure

**Response:**
1. Bootstrapper.launch() fails
2. Catch error, set failed = true
3. LPs claim refunds
4. Retry when Uniswap recovers

### Scenario 4: Large-Scale Exploit

**Response:**
1. Guardian emergency pause
2. Stop new launches
3. Audit all active launches
4. Communicate with LPs
5. Plan fix/mitigation
6. Gradual recovery

## Scaling Considerations

### Capacity Estimates (Single Network)

| Metric | Estimate |
|--------|----------|
| Block gas limit | 30M (Base), 100M (Arbitrum) |
| Deposits/block | 50-100 (at ~80k gas each) |
| Launches/block | 1-2 (at ~500k gas each) |
| Governance votes/block | 50-100 (at ~50k gas each) |

### Theoretical Maximum

```
Base (30M gas/block, 2s blocks):
- ~450 deposits per second
- ~1.5 launches per second
- ~150 votes per second

Arbitrum (100M gas/block, 0.5s blocks):
- ~500 deposits per second
- ~2 launches per second
- ~300 votes per second
```

### When to Expand

Consider v2 deployment when:
- Total active launches > 500
- Daily transaction volume > 100k
- Average block utilization > 50%
- User complaints about gas/congestion
